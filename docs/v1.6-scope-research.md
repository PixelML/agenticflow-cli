## Cross-project dependencies

### ccav producer relationship

Source: [CROSS-PROJECT.md](/Users/sean/WIP/Antigravity-Workspace/pixelml/6_Technical/CROSS-PROJECT.md#L10)

Locked dependency:
- `ccav v1.1 P9` produces alert-threshold patches (`.json`) for `ccafcli v1.6 video-action workflow`.

Implication for v1.6:
- The video-action workflow phase should not invent thresholds locally.
- It should consume ccav’s threshold artifact format as an external input and keep the CLI side focused on packaging, routing, and workflow invocation.

### ccishi remote-driver pattern

Sources:
- [docs/phase11-telegram-driver-research.md](/Users/sean/WIP/Antigravity-Workspace/ishi-core/docs/phase11-telegram-driver-research.md)
- [docs/p10-gap-closure-patch.md](/Users/sean/WIP/Antigravity-Workspace/ishi-core/docs/p10-gap-closure-patch.md)
- [CROSS-PROJECT.md](/Users/sean/WIP/Antigravity-Workspace/pixelml/6_Technical/CROSS-PROJECT.md#L11)

Useful adoption idea for future ccafcli remote-CLI use:
- keep transport adapters thin
- use typed protocol events instead of ad hoc chat text
- prefer single-writer outbound queues with replay sequencing for streamed replies/progress updates
- keep operator secrets/config at the server boundary, not in per-session caller state

This is not a v1.6 must-have unless Phase 11 above is chosen, but it is the right pattern if `af gateway` evolves into a richer remote agent driver surface.

## Risks to watch

- v1.5 verification is complete on code, but still `human_needed` on real platform behavior for truncation, GitHub catalog smoke tests, and company export/import round-trip. Those are not theoretical risks; they are explicitly still pending.
- Company import/export is now a public YAML contract via [packages/cli/src/cli/company-io.ts](/Users/sean/WIP/Antigravity-Workspace/agenticflow-js-cli/packages/cli/src/cli/company-io.ts). Any v1.6 work touching it needs to preserve backward compatibility.
- Platform catalog depends on GitHub Tree API and parallel raw fetches in [packages/cli/src/cli/platform-catalog.ts](/Users/sean/WIP/Antigravity-Workspace/agenticflow-js-cli/packages/cli/src/cli/platform-catalog.ts); rate limits and malformed pack files are already handled, but still fragile relative to first-party platform APIs.
- The command surface is concentrated in [packages/cli/src/cli/main.ts](/Users/sean/WIP/Antigravity-Workspace/agenticflow-js-cli/packages/cli/src/cli/main.ts), which is already very large. v1.6 phases should keep extracting modules instead of deepening the monolith.
- `.planning/PROJECT.md` still records pre-existing failures in `main.test.ts` around agent clone/usage/chat/workflow-watch. If v1.6 touches adjacent code paths, those failures will make regression attribution harder.
